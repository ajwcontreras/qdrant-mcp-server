The original RRF paper fixed k=60 during pilot experiments and showed RRF often outperformed individual rankers and Condorcet/CombMNZ, but it was not code-specific. SciSpace +1 For your case, tune k on your 240 golden queries. Try: k ∈ {5, 10, 20, 30, 60, 100} BM25 weight ∈ {0.8, 1.0, 1.2, 1.5, 2.0} vector weight ∈ {0.8, 1.0, 1.2, 1.5, 2.0} Because your vector channel has high recall but poor ordering, I would start with lower k values plus BM25 upweighting. Lower k gives more influence to top-ranked items; higher k makes deeper ranks matter more. Elastic’s hybrid-search analysis notes that k=60 is inherited from the RRF paper, not universally optimal, and that retrieval depth also affects final quality. elastic.co Alternatives worth testing: Weighted RRF: best low-risk change because it keeps rank robustness while correcting channel imbalance. Convex combination / linear fusion: can outperform RRF when scores are calibrated, but is sensitive to score normalization. Bruch et al. compare convex combination and RRF and discuss the pitfalls of hybrid fusion. arXiv Relative score fusion: normalize BM25 and vector scores to 0–1 before weighted summation; Weaviate uses configurable hybrid fusion and weighting. Weaviate Documentation +1 Distribution-Based Score Fusion: useful when BM25 and vector score distributions vary by query; Qdrant-style DBSF normalizes by score distribution rather than raw min/max. Prem AI Action: do not assume k=60. Grid-search fusion parameters by query type. For symbol/debugging queries, bias lexical/BM25; for architectural/data-flow queries, bias vector. 2. Reranking for code search A reranker is the highest-leverage fix for your MRR problem. Cross-encoders score the query and candidate together, which is slower than embeddings but much better for final ordering; BGE’s own model card describes the standard pattern: retrieve top 100, rerank them, return the top few. Hugging Face bge-reranker-base is usable but has a 512-token limit, so it will work best if chunks are function-sized or aggressively trimmed. Superlinked For code, consider testing: Qwen3-Reranker-0.6B / 4B: Qwen3 explicitly targets embedding and reranking across text and code retrieval. arXiv +1 Jina reranker v2: Jina reports CodeSearchNet reranking benchmarks and explicitly evaluates code retrieval MRR@10. Jina AI bge-reranker-v2-m3 if you need longer context than 512 tokens; discussions note it extends beyond the classic XLM-R 512-token limit. Hugging Face Expected impact: for your system, I would expect large MRR gains because Recall@10 is already high. The reranker cannot recover documents absent from candidates, but your candidates already contain the right file 92.1% of the time. A realistic target is moving MRR from 0.476 toward 0.60–0.70, especially if reranking top 50–100. Action: retrieve top 100 from hybrid, rerank top 50–100 with cross-encoder, evaluate MRR by query type. Use smaller chunks or “rerank text” composed of signature + docstring + identifiers + surrounding comments + first N logical lines. 3. Chunking strategy for code Naive 1500-char line chunks are likely hurting both embeddings and reranking. Recent code-RAG work strongly favors AST-aware chunking. CAST proposes chunking via Abstract Syntax Trees, recursively splitting large AST nodes and merging siblings to preserve semantic coherence; it reports gains over fixed-size chunking on retrieval and downstream code-generation tasks. CMU School of Computer Science +1 Glean’s code search docs also describe AST-based chunking into meaningful chunks such as imports, related functions, and classes, specifically to avoid splitting functions in half. Glean Documentation Recommended chunk hierarchy: Primary unit: function/method. Class chunks: class signature + fields + method summaries, not entire giant classes unless small. Import/config chunks: keep imports, routes, Flask blueprints, decorators, environment config as special chunks. Context envelope: prepend file path, module name, class name, function signature, decorators, docstring, and nearby comments. Large functions: split by AST blocks but repeat function signature and parent context. Small related functions: merge siblings up to token budget. Action: replace line chunking with tree-sitter/AST chunking. Target 300–900 tokens for embedding chunks and ≤512 tokens for bge-reranker-base rerank text. Keep file-level aggregation for final display. 4. HyDE for code Your observation is consistent with HyDE’s design: HyDE creates a hypothetical document from the query and embeds that synthetic document to bridge query-document mismatch. The original HyDE paper describes it as a query-time pivot through generated hypothetical documents, with the dense encoder acting as a bottleneck that filters incorrect generated details. arXiv +1 But prepending 12 generated questions to every code chunk can cause embedding dilution: the vector no longer represents code only; it represents code plus synthetic natural language. That can help broad semantic queries and hurt debugging/exact-match queries. Better architecture: Keep code embeddings separate. Keep HyDE/question embeddings in a separate index or separate named vector. Fuse at query time. Generate HyDE per query, not only per chunk, for architectural/behavioral queries. For debugging/symbol queries, disable HyDE or heavily downweight it. Instead of only generating questions, generate multiple code-specific representations: natural-language summary API/function intent symbol list error/debug scenarios call relationships “when would I use this?” description This is closer to multi-representation retrieval, which is well supported in dense retrieval literature; multi-representation methods preserve different relevance views rather than forcing all semantics into one vector. arXiv +1 Action: stop prepending HyDE into code content. Store code_dense, summary_dense, hyde_question_dense, and identifier_sparse separately, then fuse by query type. 5. Embedding model selection for code Use code-specific benchmarks, not only MTEB. GitHub +1 Known options: Qwen3-Embedding: explicitly designed for text retrieval, code retrieval, and reranking. Hugging Face +1 BGE-M3: supports dense, sparse, and multi-vector retrieval in one model family. Measure Recall@10, MRR, nDCG@10 by query type. 6. BM25 tokenization for code Trigram helps partial matching, but it is not enough. Source code has meaningful structure: identifiers, paths, symbols, decorators, route names, class names, and API names. The source-code-search survey notes that many code search methods tokenize by splitting camelCase and snake_case, and that BM25 remains widely used in code search. Weisong Sun's Homepage Research and practice support identifier splitting. Code summarization/retrieval pipelines commonly split CamelCase and snake_case so models can learn subtoken semantics. arXiv +1 Sourcegraph reported adapting BM25 into BM25F for code search and improving code search relevance by weighting fields differently. Sourcegraph Recommended BM25 fields: identifier_exact: function/class/variable names, high boost identifier_subtokens: camel/snake split, high boost path: file path/module path, high boost signature: function/class signature, high boost decorators/routes: Flask routes, CLI commands, event handlers comments_docstrings: medium boost body_tokens: lower boost trigrams: fallback for partial/typo/symbol fragments Action: move from one trigram FTS field to BM25F-style multi-field search. For symbol lookup, exact identifier/path matches should dominate dense similarity. 7. Query understanding / rewriting Query rewriting can help code search, but only if it is intent-aware. Generic expansion can hurt debugging and symbol queries by washing out precise terms. General RAG guidance distinguishes expansion, decomposition, and multi-query retrieval; Anyscale’s retrieval guide recommends query rewriting/expansion, multi-query retrieval, and decomposition for different query shapes. Anyscale Docs Use a lightweight classifier first: symbol_lookup: no rewrite; preserve exact tokens; add camel/snake variants. debugging: preserve error text, stack traces, route names, and variable names; maybe add likely synonyms. architectural: multi-query expansion helps. data_flow: decompose into source, transformation, sink. integration: expand API/library/framework terms. behavioral: rewrite into “where is behavior implemented?” Recent query-understanding work also supports decomposition plus semantic interpretation before fusing retrieval results. UvA FNWI Staff Action: do not enable one-size-fits-all LLM rewriting. Add query-type routing and evaluate per type. Preserve original query as one retrieval branch and fuse rewritten/decomposed branches with it. 8. Multi-vector vs single-vector + BM25 Multi-vector is genuinely useful when representations capture different relevance signals. BGE-M3 is a good example: it is designed to support dense retrieval, sparse retrieval, and multi-vector retrieval simultaneously. GitHub ColBERT-style multi-vector retrieval also improves fine-grained matching by representing documents as sets of token vectors rather than one pooled vector, though it is more expensive. GitHub +1 But your old 4-vector Qdrant architecture is not automatically better. Multi-vector helps when each vector has a clear job: code_dense: implementation semantics summary_dense: natural-language intent hyde_dense: query/document mismatch bridge lexical_sparse: exact terms and identifiers Your current Vectorize+D1 result already proves that single-vector + BM25 can beat a multi-vector setup on recall if chunking, embeddings, and fusion are better. The issue is ranking, not candidate discovery. Action: use multi-representation selectively, not blindly. Rebuild multi-vector as separate retrievable channels with learned/tuned weights by query type, then rerank. Concrete roadmap Phase 1: Ranking fix Retrieve top 100 from vector and top 100 from BM25.
